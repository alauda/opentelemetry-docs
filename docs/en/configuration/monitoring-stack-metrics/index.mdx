# Configuring Metrics for the Monitoring Stack

As a cluster administrator, you can configure the OpenTelemetry Collector to integrate with the Alauda Container Platform monitoring stack. This integration enables you to:

- Create Prometheus `ServiceMonitor` custom resources for scraping the Collector's pipeline metrics and enabled Prometheus exporters
- Configure the Prometheus receiver to scrape metrics from the in-cluster monitoring stack

## Sending Metrics to the Monitoring Stack

You can configure the `OpenTelemetryCollector` custom resource (CR) to create a Prometheus `ServiceMonitor` CR or a `PodMonitor` CR for a sidecar deployment. A `ServiceMonitor` can scrape the Collector's internal metrics endpoint and Prometheus exporter metrics endpoints.

### ServiceMonitor Configuration

The following example shows an OpenTelemetry Collector CR configured with the Prometheus exporter:

```yaml
apiVersion: opentelemetry.io/v1beta1
kind: OpenTelemetryCollector
spec:
  mode: deployment
  observability:
    metrics:
      enableMetrics: true # 1
  config:
    exporters:
      prometheus:
        endpoint: 0.0.0.0:8889
        resource_to_telemetry_conversion:
          enabled: true # by default resource attributes are dropped
    service:
      telemetry:
        metrics:
          readers:
          - pull:
              exporter:
                prometheus:
                  host: 0.0.0.0
                  port: 8888
      pipelines:
        metrics:
          exporters: [prometheus]
```

**Callout 1**: Configures the Alauda build of OpenTelemetry Operator to create the Prometheus `ServiceMonitor` CR or `PodMonitor` CR to scrape the Collector's internal metrics endpoint and the Prometheus exporter metrics endpoints.

:::note
Setting `enableMetrics` to `true` creates the following two `ServiceMonitor` instances:

- One `ServiceMonitor` instance for the `<instance_name>-collector-monitoring` service. This `ServiceMonitor` instance scrapes the Collector's internal metrics.
- One `ServiceMonitor` instance for the `<instance_name>-collector` service. This `ServiceMonitor` instance scrapes the metrics exposed by the Prometheus exporter instances.
:::

### PodMonitor Configuration for Sidecar Deployments

Alternatively, a manually created Prometheus `PodMonitor` CR can provide fine control, for example removing duplicated labels added during Prometheus scraping.

The following example shows a `PodMonitor` CR that configures the monitoring stack to scrape the Collector metrics:

```yaml
apiVersion: monitoring.coreos.com/v1
kind: PodMonitor
metadata:
  name: otel-collector
spec:
  selector:
    matchLabels:
      app.kubernetes.io/name: <cr_name>-collector # 1
  podMetricsEndpoints:
  - port: metrics # 2
  - port: promexporter # 3
    relabelings:
    - action: labeldrop
      regex: pod
    - action: labeldrop
      regex: container
    - action: labeldrop
      regex: endpoint
    metricRelabelings:
    - action: labeldrop
      regex: instance
    - action: labeldrop
      regex: job
```

**Callout 1**: The name of the OpenTelemetry Collector CR.

**Callout 2**: The name of the internal metrics port for the OpenTelemetry Collector. This port name is always `metrics`.

**Callout 3**: The name of the Prometheus exporter port for the OpenTelemetry Collector.
