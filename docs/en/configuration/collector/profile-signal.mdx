# Profile Signal

The Profile signal represents a new telemetry data type designed to capture detailed information about code execution patterns and resource consumption within applications.

:::warning
The Profile signal is a Technology Preview feature.
:::

By leveraging the Profile signal, you can identify performance bottlenecks and resource inefficiencies at the function level, down to specific lines of code. When combined with traces, metrics, and logs, profiling data provides a holistic view of application behavior, enabling precise performance analysis and targeted optimization in production environments.

## Profiling Targets

Profiling can be applied at two distinct levels:

- **Application profiling**: Helps developers assess code performance, catch regressions early, and track resource consumption patterns such as CPU and memory usage, making it easier to locate and optimize inefficient code paths.
- **Operating system profiling**: Offers visibility into lower-level system behavior, including system calls, kernel operations, and I/O wait times, which can inform infrastructure tuning and cost optimization efforts.

```yaml
apiVersion: opentelemetry.io/v1beta1
kind: OpenTelemetryCollector
metadata:
  name: otel-profiles-collector
  namespace: otel-profile
spec:
  args:
    feature-gates: service.profilesSupport # [!code callout]
  config:
    receivers:
      otlp: # [!code callout]
        protocols:
          grpc:
            endpoint: "0.0.0.0:4317"
          http:
            endpoint: "0.0.0.0:4318"
    exporters:
      otlp/pyroscope: # [!code callout]
        endpoint: "pyroscope.pyroscope-monitoring.svc.cluster.local:4317"
    service:
      pipelines: # [!code callout]
        profiles:
          receivers: [otlp]
          exporters: [otlp/pyroscope]
# ...
```

<Callouts>
1. Enables the profiles feature by setting the `feature-gates` argument to `service.profilesSupport`.
2. Configures the OTLP Receiver so that the Collector can accept profile data over gRPC and HTTP.
3. Specifies the export destination for profile data, such as an OTLP-compatible profiling backend.
4. Defines a `profiles` pipeline that routes received profile data to the configured exporter.
</Callouts>
