# Debug Exporter

The Debug Exporter is a specialized exporter for troubleshooting that outputs collected telemetry data to standard output (stdout), allowing you to directly view the data content received and processed by the Collector.

## Overview

The Debug Exporter is primarily used in the following scenarios:

- Verifying whether the Collector correctly receives telemetry data
- Checking whether data format and content meet expectations
- Debugging data processing pipeline issues
- Quickly viewing data in development and testing environments

:::warning
The Debug Exporter outputs all data to logs, which can generate substantial log volume. It is not recommended for long-term use in production environments.
:::

## Configuring Debug Exporter

You can configure the Debug Exporter in the `OpenTelemetryCollector` custom resource:

```yaml
config:
  exporters:
    debug:
      verbosity: detailed
  service:
    pipelines:
      traces:
        exporters: [debug]
      metrics:
        exporters: [debug]
      logs:
        exporters: [debug]
```

**Configuration parameters**

- `verbosity`: Output detail level, options:
  - `basic`: Basic information
  - `normal`: Standard information (default)
  - `detailed`: Detailed information, including complete data content

## Viewing exported data

After configuring the Debug Exporter, exported data appears in the Collector pod logs. View them using the following command:

```bash
kubectl logs <collector-pod-name> -n <namespace> -f
```

**Example output**

Traces output example:

```
2024-03-12T10:30:45.123Z	info	debugexporter/debug_exporter.go:40	TracesExporter	{"kind": "exporter", "data_type": "traces", "name": "debug", "resource spans": 1, "spans": 3}
```

Metrics output example:

```
2024-03-12T10:30:46.456Z	info	debugexporter/debug_exporter.go:50	MetricsExporter	{"kind": "exporter", "data_type": "metrics", "name": "debug", "resource metrics": 1, "metrics": 5, "data points": 10}
```

## Using Debug Exporter with other exporters

The Debug Exporter can be used simultaneously with other exporters, allowing you to send data to backend systems while viewing data content in logs:

```yaml
config:
  exporters:
    debug:
      verbosity: detailed
    otlp:
      endpoint: jaeger-collector:4317
      tls:
        insecure: true
  service:
    pipelines:
      traces:
        receivers: [otlp]
        processors: [batch]
        exporters: [debug, otlp]
```

In this configuration, traces data is sent to both the Debug Exporter and the OTLP exporter.

## Filtering debug output

If the Debug Exporter produces excessive data output, you can use processors to filter data:

```yaml
config:
  processors:
    filter/debug:
      traces:
        span:
          - 'attributes["http.status_code"] >= 400'
  exporters:
    debug:
      verbosity: detailed
  service:
    pipelines:
      traces:
        receivers: [otlp]
        processors: [filter/debug]
        exporters: [debug]
```

This configuration only outputs spans with HTTP status codes greater than or equal to 400 to the Debug Exporter.

## Troubleshooting with Debug Exporter

**Verifying data reception**

If you suspect the Collector is not receiving data, enabling the Debug Exporter can quickly confirm:

1. Configure the Debug Exporter and set `verbosity: detailed`
2. View the Collector logs
3. If no data output appears in the logs, the Collector is not receiving data

**Checking data format**

Use the Debug Exporter to view complete data content and confirm:

- Whether the data format is correct
- Whether required attributes are present
- Whether data values meet expectations

**Debugging processors**

Add Debug Exporters before and after processors to compare data changes:

```yaml
config:
  exporters:
    debug/before:
      verbosity: detailed
    debug/after:
      verbosity: detailed
  service:
    pipelines:
      traces:
        receivers: [otlp]
        processors: []
        exporters: [debug/before]
      traces/processed:
        receivers: [otlp]
        processors: [batch, attributes]
        exporters: [debug/after]
```

## Performance considerations

The Debug Exporter impacts performance, especially in high-traffic scenarios:

- Outputting detailed logs consumes CPU and memory
- Large log volumes may lead to disk space exhaustion
- Log writing may become a performance bottleneck

:::tip
In production environments, it is recommended to enable the Debug Exporter only temporarily when needed, and disable it immediately after troubleshooting is complete.
:::

## Disabling Debug Exporter

After completing troubleshooting, remove the Debug Exporter from the pipeline configuration:

```yaml
config:
  service:
    pipelines:
      traces:
        receivers: [otlp]
        processors: [batch]
        exporters: [otlp]  # Remove debug
```

After applying the configuration, the Collector automatically reloads the configuration and stops outputting debug logs.
