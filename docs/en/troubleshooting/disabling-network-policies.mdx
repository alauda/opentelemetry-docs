# Disabling network policies

The Alauda Build of OpenTelemetry v2 Operator creates network policies to control traffic for the Operator and Collector components, enhancing security. By default, network policies are enabled and configured to allow traffic between all required components, requiring no additional configuration.

## When to disable network policies

In certain situations, you may need to disable network policies to troubleshoot issues:

- The OpenTelemetry Collector cannot receive telemetry data
- The Collector cannot export data to backend systems
- The Target Allocator component experiences connection problems
- You need to verify whether network policies are causing traffic issues

:::warning
Disabling network policies reduces cluster security. Only disable temporarily during troubleshooting, and re-enable immediately after resolving the issue.
:::

## Understanding network policies

Network policies created by the Alauda Build of OpenTelemetry v2 Operator include:

**Collector network policies**

- Allow inbound traffic from application pods to the Collector
- Allow outbound traffic from the Collector to backend systems
- Allow communication between Collector internal components

**Target Allocator network policies**

- Allow the Collector to access the Target Allocator
- Allow the Target Allocator to access the Kubernetes API
- Allow the Target Allocator to discover and scrape targets

## Disabling network policies for the Collector

You can disable network policies for the Collector by configuring the `OpenTelemetryCollector` custom resource:

```yaml
apiVersion: opentelemetry.io/v1beta1
kind: OpenTelemetryCollector
metadata:
  name: otel
  namespace: opentelemetry-collector
spec:
  networkPolicy:
    enabled: false
  config:
    receivers:
      otlp:
        protocols:
          grpc: {}
          http: {}
    exporters:
      otlp:
        endpoint: jaeger-collector:4317
    service:
      pipelines:
        traces:
          receivers: [otlp]
          exporters: [otlp]
```

**Configuration description**

- `networkPolicy.enabled`: Set to `false` to disable network policies
- When disabled, the Collector is not restricted by network policies and can communicate with any pod

## Verifying network policy status

After applying the configuration, verify that network policies are disabled:

```bash
kubectl get networkpolicy -n opentelemetry-collector
```

If network policies are disabled, you should not see NetworkPolicy resources related to the Collector.

## Troubleshooting network connectivity

If issues persist after disabling network policies, check other possible causes:

**Check Service configuration**

Verify that the Collector Service is correctly configured:

```bash
kubectl get svc -n opentelemetry-collector
kubectl describe svc <collector-service-name> -n opentelemetry-collector
```

**Check port configuration**

Confirm that the ports the Collector listens on match the ports exposed by the Service:

```bash
kubectl get pod <collector-pod-name> -n opentelemetry-collector -o jsonpath='{.spec.containers[0].ports}'
```

**Test connectivity**

Test connectivity from an application pod to the Collector:

```bash
kubectl exec -it <app-pod-name> -n <app-namespace> -- curl -v http://<collector-service>:4318/v1/traces
```

**Check firewall rules**

If the cluster uses additional firewalls or security groups, confirm they are not blocking traffic.

## Re-enabling network policies

After completing troubleshooting, re-enable network policies to restore security:

```yaml
apiVersion: opentelemetry.io/v1beta1
kind: OpenTelemetryCollector
metadata:
  name: otel
  namespace: opentelemetry-collector
spec:
  networkPolicy:
    enabled: true
  # ... other configuration
```

After applying the configuration, the Operator automatically recreates the network policies.

## Customizing network policies

If the default network policies do not meet your requirements, you can customize network policy configuration:

```yaml
apiVersion: opentelemetry.io/v1beta1
kind: OpenTelemetryCollector
metadata:
  name: otel
  namespace: opentelemetry-collector
spec:
  networkPolicy:
    enabled: true
    ingress:
      - from:
        - namespaceSelector:
            matchLabels:
              name: my-app-namespace
        ports:
        - protocol: TCP
          port: 4317
        - protocol: TCP
          port: 4318
    egress:
      - to:
        - namespaceSelector:
            matchLabels:
              name: observability-backend
        ports:
        - protocol: TCP
          port: 4317
```

This configuration allows:

- Inbound traffic from the `my-app-namespace` namespace to ports 4317 and 4318
- Outbound traffic to port 4317 in the `observability-backend` namespace

## Best practices

**Troubleshooting workflow**

1. First check Collector logs and metrics
2. Verify Service and port configuration
3. If network policies are suspected, temporarily disable them
4. If disabling resolves the issue, it indicates a network policy configuration problem
5. Customize network policies based on actual requirements
6. Re-enable network policies and verify

**Security recommendations**

- Only disable network policies when necessary
- Closely monitor cluster security status while disabled
- Re-enable network policies as soon as possible
- Use the principle of least privilege when configuring custom network policies

:::note
If your cluster uses other network policy controllers (such as Calico or Cilium), ensure they are compatible with Kubernetes NetworkPolicy.
:::
